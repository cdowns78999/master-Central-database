What even IS this?

You're about to play with a pattern that went viral in April 2026: Andrej Karpathy's "LLM Wiki". Instead of stuffing raw documents into a context window or a vector database, Claude reads your sources and compiles them into a cross-linked markdown wiki — an organized, compounding knowledge base you can actually read.

This lab adds one twist: a Pinecone vector database layer underneath for deep semantic search when the wiki alone isn't enough. That's the hybrid your dad was describing — a small curated wiki on top, a deep vector index below.

The three layers you're building:
  • sources/ — raw files you drop in (transcripts, PDFs, notes)
  • wiki/ — the compiled markdown articles Claude writes
  • Pinecone — the semantic search index for deep recall

Folder tour

Everything lives in one sealed folder. Nothing else on your computer gets touched.

📂 Karpathy Wiki Lab/ ├── 🚀 claude.ps1 ← main launcher (PowerShell) ├── 🖱 claude.bat ← double-click wrapper ├── 📄 CLAUDE.md ← the rulebook ├── 📁 sources/ ← drop raw files here ├── 📁 wiki/ ← Claude writes articles here ├── 📁 logs/ ← every action is logged ├── 📁 .backups/ ← auto-snapshots before changes └── 🌐 guide.html ← you're reading it

Each folder has a single, clean job. The .backups/ folder is your safety net — every time you run the indexer, it snapshots the current wiki state first.

Run the launcher

Double-click claude.bat in the lab folder. A terminal window opens, you'll see a banner, then a menu. No typing commands — just press a number.

+--------------------------------------+ | MENU | +--------------------------------------+ | [1] Health check | | [2] Dry-run (no writes) | | [3] Index sources -> wiki | | [4] View latest log | | [5] Pinecone setup (login + API) | | [6] Open interactive guide | | [q] Quit | +--------------------------------------+

First time? Press 1 to run the health check. It verifies all folders exist and the schema is readable. Every action gets written to logs/.

Safety: Option 2 (Dry-run) previews what WOULD happen without writing a single file. Use it anytime you're nervous.

Pinecone setup (login + API key)

Pinecone is the vector database — it's where your embeddings live so the lab can do semantic search across thousands of chunks. The free tier is robust enough for a personal niche: one Starter index, roughly 100K vectors, serverless (no pod management).

  1. Sign up at app.pinecone.io — email or Google auth
  2. Create a Starter index — name: karpathy-lab, dimensions: 1536, metric: cosine
  3. Grab your API key at app.pinecone.io/keys
  4. Back in the launcher, press [5] and paste the key when prompted
Security: The API key is stored locally in .pinecone-config inside the lab folder. Don't commit that file to git or share the folder zipped.

5

Drop your first document

Time to give Claude some raw material. Open sources/ and drop in any file — a transcript, a PDF, a .txt dump, an exported article. Markdown works great too.

Pro move: Create a file called sources/NICHE.md first. Put one paragraph describing what niche this lab is focused on. Claude will use it as the filter — "what matters, what to ignore." Without it, the wiki sprawls.

Example NICHE.md:

This lab is scoped to music marketing campaigns for independent artists. Topics of interest: playlist pitching, TikTok strategy, ad spend ratios, PR outreach, release timing. Ignore anything unrelated.

6

Watch Claude index it

Back in the launcher, press [3] Index sources → wiki. The script backs up the current wiki to .backups/TIMESTAMP/ then signals Claude to process every file in sources/.

Claude reads each source, follows the rules in CLAUDE.md, and writes organized markdown articles into wiki/. Each article has frontmatter, a summary, key points, and cross-links to related articles.

wiki/ ├── playlist-pitching.md ├── tiktok-strategy.md ├── release-timing.md └── ad-spend-ratios.md

Over time, the wiki compounds. New sources merge into existing articles instead of creating duplicates. That's the Karpathy insight — the artifact grows smarter the more you feed it.

7

Query, iterate, debug

Your wiki is now a real, searchable knowledge base. Two ways to pull info:

  • Open a wiki article directly — they're just markdown files, readable anywhere
  • Ask Claude a semantic question — Claude uses Pinecone to retrieve the top-K most relevant chunks, then answers using the wiki article as anchor context
Debug loop when something looks off:
  1. Launcher → [4] View latest log — see every action that ran
  2. Check .backups/ — previous state is always recoverable
  3. Tweak CLAUDE.md if article structure needs fixing
  4. Re-run [3] Index — the wiki recompiles against new rules

The whole point: it's a feedback loop you own. Dad was right — the tuning is in how you chunk, what you index, what CLAUDE.md tells Claude to prioritize. Keep iterating.
