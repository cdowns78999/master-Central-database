Automation Patterns

The 7 proven architectures for overnight AI

01
The Ralph Loop
while true; do claude-code --prompt task.md; done
The simplest and most battle-tested pattern. An infinite loop that feeds a task file to Claude Code on repeat. Each iteration picks up where the last left off. Named after Geoffrey Huntley's Ralph Wiggum loop that ran for 3 months straight. The philosophy is persistent iteration — brute-force consistency that compounds over time.
Best for: batch operations, large refactors, test coverage expansion, repetitive file processing
Persistent
02
Continuous PR Loop
Branch → Code → Commit → Push → PR → CI → Merge → Repeat
A structured loop that creates proper git branches, writes code, commits, pushes, opens a pull request, waits for CI, and merges on success. A SHARED_TASK_NOTES.md file preserves context between iterations so the agent remembers what it did and what comes next. Each PR is atomic and reviewable.
Best for: multi-step refactors, incremental feature development, teams that need audit trails
Structured
03
Cron-Triggered Agent
0 2 * * * /usr/bin/claude-agent run nightly-review
Schedule-based one-shot execution. Set it via crontab, launchd, Windows Task Scheduler, or GitHub Actions cron. The agent wakes at a fixed time (typically 2AM), performs its task, saves a report, and exits cleanly. No loops, no state management. Each run is independent.
Best for: nightly code reviews, documentation generation, security scans, dependency audits
Scheduled
04
Heartbeat Agent
wake(interval) → check_queue() → execute() → report() → sleep()
A long-running daemon that wakes at configurable intervals (typically hourly). It checks a task queue or inbox for new work, executes any pending tasks, writes a status report, then goes back to sleep. Long-term memory persists across cycles via a local context file. Think of it as a dedicated overnight employee.
Best for: always-on assistants, inbox monitoring, task queues, long-running research
Daemon
05
Multi-Agent Swarm
lead.assign(specialists) → parallel_execute() → lead.merge(results)
A team lead agent coordinates specialist agents, each running in their own context window and git worktree. One agent refactors, another writes tests, a third handles docs, a fourth does security review. The lead assigns work, monitors progress, and merges results. This is the overnight factory floor.
Best for: large-scale overnight work, full-stack rewrites, multi-concern codebases
Parallel
06
GitHub Actions Pipeline
on: [schedule, issues, pull_request] → @claude → execute
Triggered by schedule, issue creation, or PR events inside GitHub Actions. Use @claude mentions in issues or comments to trigger work. The agent runs in CI, has access to the full repo, and can create PRs, review code, or close issues. Fully cloud-hosted — no local machine needed.
Best for: team workflows, CI/CD integration, automated code review, issue triage
Cloud
07
Slack-Driven Queue
slack_cmd → sqlite_queue → agent_daemon → claude_executor
Team members submit tasks via a Slack command. Tasks go into a SQLite queue with priority levels. An agent daemon monitors the queue and dispatches work to Claude. Day/night cost thresholds control spend — cheap tasks run immediately, expensive ones queue for off-peak. Results post back to Slack.
Best for: team task management, cost optimization, async workflows, non-technical stakeholders
Queue

What The Data Tells Us

5 insights from hundreds of overnight agent runs

Insight 01
Consistency > Intensity
Nightly 8-hour runs consistently outperform 16-hour weekend sprints. The compounding effect of daily incremental progress creates exponentially better outcomes than burst effort. The tortoise wins every time — especially when the tortoise never sleeps.
Insight 02
TDD Is Non-Negotiable
Overnight agents without test validation create net-negative cleanup work. Every successful overnight operation has test-driven development baked in. The agent writes the test first, then the code, then validates. No exceptions. Without TDD, you wake up to a mess.
Insight 03
Context Is King
SHARED_TASK_NOTES.md between iterations is what separates good runs from wasted API costs. Without persistent context, each agent invocation starts cold. With it, the agent knows its history, its failures, and its plan. Context files are the overnight agent's memory.
Insight 04
Trust Calibration
95% accuracy per step equals chaos over 20-step workflows. If each step has a 5% failure rate, the probability of a clean 20-step run is just 36%. Design the handoff, not the agent. Human checkpoints at critical junctures are the architecture — not the bottleneck.
Insight 05
Guardrails > Guidance
Rate limiting, CI gates, and PR reviews are your safety net. Not prompts. You cannot prompt your way to overnight reliability. You build it with hard infrastructure: test suites that block merges, cost caps that kill runaway agents, and review gates that require approval. Guardrails are the architecture of trust.
